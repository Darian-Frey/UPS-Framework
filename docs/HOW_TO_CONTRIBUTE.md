# Contributing to the UPS-Framework
**Protocol:** SCHEMA_V5 | **Framework:** Universal Prismatic Stability V1.1

Welcome, Researcher. The UPS-Framework is a collaborative environment dedicated to solving high-complexity problems through the lens of **Prismatic Stability**. Because this project bridges p-adic geometry and computer science, we maintain rigorous standards for all contributions.

---

## 🛠 1. Core Principles

### A. Prismatic Integrity
All mathematical contributions must be evaluated against the **Prismatic Discriminant ($\Delta_{pris}$)**. We do not accept "vibe-based" proofs. If you propose a new obstruction or a motivic lift, you must provide the stability slope analysis.

### B. Multi-Agent Verification (The "Cluster" Rule)
This repository was built using a multi-agent AI cluster. If you are using AI to assist your research, you must disclose the model and the specific role it played (e.g., "GCT Auditor," "Lean 4 Formalizer"). 

### C. One-In, One-Out Workflow
To prevent architectural drift, we adhere to a strict sequential task assignment. Do not open multiple massive Pull Requests (PRs) simultaneously. One task must be verified and merged before the next is assigned.

## 🧪 2. How to Contribute

### Reporting Ghosts (Issues)
If you find a potential "Leakage" in our stability walls (e.g., a case where $m_\lambda(\text{det}_n) > 0$ for $h \ge 6$), open an Issue titled `GHOST_REPORT: [Description]`.
* Provide the Partition $\lambda$.
* Provide the dimension $n$.
* Provide the citation or numerical log.

### Submitting Stability Tests (Pull Requests)
We welcome new stress tests for the `research/` directories.
1. **Fork** the repository.
2. **Create a branch** named `feature/stability-[problem-name]`.
3. **Include a README** in your new sub-directory explaining the Prismatic logic.
4. **Run the script** and paste the terminal output into the PR description as "Proof of Stability."

## 📐 3. Technical Standards

* **Language:** Python 3.10+ (Numerical), Lean 4 (Formal Proofs), Markdown (Documentation).
* **Formatting:** Use LaTeX for all formal math. Use the `D_TASK` structure for any proposed research sprints.
* **Code Style:** Prioritize clarity and symbolic accuracy over raw performance. We are mapping varieties, not mining coins.

## 📜 4. Contributor Covenant
By contributing to this repo, you agree to maintain the "Source of Truth" established in the `ROM` files. We value candor and precision. If a theory is proven wrong by a stress test, we acknowledge the "Ghost" and realign the framework.
